**What you're actually describing is two separate things:**

1. **Procedural knowledge** — *how to do things* → that's Skills (already solved)
2. **Declarative identity** — *who the agent is, what it knows, who it serves* → that's the gap

Right now our "identity" is tied to OpenClaw's workspace runtime. The question is: how do you make it portable?

---

**The cleanest architecture I see:**

**Layer 1 — Context Package (static)**
A self-contained folder that any runtime can consume:
```
agent-context/
  identity.md     ← who I am
  soul.md         ← how I think, values, style
  user.md         ← who I'm helping
  memory.md       ← long-term knowledge
  studio.md       ← org context (laniameda-hq)
  README.md       ← how to inject this into a system prompt
```

This is basically what we already have, but extracted from OpenClaw and made portable. Any system that reads files can use it.

**Layer 2 — System Prompt Builder (dynamic)**
A script that takes the context package + a role parameter and outputs an injection-ready system prompt:
```
build-prompt.py --agent persey --layers identity,user,studio
```
Use it in Claude Desktop, Cursor, a Chrome extension, anything.

**Layer 3 — MCP Server (queryable)**
Instead of static injection, expose the context as tools:
- `get_identity()` — returns soul + identity
- `get_user_context()` — USER.md
- `search_memory(query)` — semantic search over memory
- `get_studio_context()` — org-level knowledge

Any MCP client connects and pulls context on demand. This is the most composable option long-term.

---

**Key design decision you need to make:**

Do you want **full identity injection** (agent behaves as Persey everywhere) or **selective context injection** (YouTube summarizer gets studio context + user prefs, but NOT Persey's CTO opinions)?

For the YouTube summarizer example — you don't want Persey. You want a thin agent with: user identity + studio taste + specific task instructions. So the context package needs to be **modular by layer**, not monolithic.

---

**How this connects to Agent Hub:**

Agent Hub becomes the **context registry**. Agents register their identity packages there. Other agents/tools query it. That's a clean architectural role for it — not just a hub for routing tasks, but a hub for agent identity and shared knowledge.
